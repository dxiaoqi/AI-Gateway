# ADR 0005：多部署路由与流式首事件前切换

- 状态：Accepted
- 日期：2026-07-15

## 背景

一个逻辑模型只绑定一个 Provider 部署时，供应商限流、网络超时或区域故障会直接变成业务失败。企业网关需要把客户端使用的稳定模型名与实际部署解耦，并在安全边界内选择备用部署。

流式响应还有额外约束：HTTP 200 和第一条 SSE 发给客户端后，响应已经开始。如果此时切换到另一个模型，客户端会收到由两个模型拼接的内容，语义、审计和 Usage 都不可信。

## 决策

1. 一个逻辑模型可以注册多个 Deployment；Deployment 包含唯一 ID、Provider、优先级和权重。
2. 先选择数字最小的健康优先级；只在同级部署内按权重随机。
3. 单个请求对每个部署最多尝试一次，并受 `maxAttempts` 总预算约束。
4. 只对 Provider 认证、无效响应、429、超时和不可用错误进行失败切换。请求校验、ACL 和企业配额错误不切换。
5. 429 进入短期冷却；超时/不可用等连续失败达到阈值后熔断。熔断期结束只允许一个半开探测。
6. 非流式响应成功后结束路由。流式调用只在取得 `response_start` 之前允许切换；之后固定在该部署，失败以 SSE error 结束。
7. 路由结果在响应元数据和结构化日志中记录 deployment ID 与尝试次数，但不记录密钥或 Prompt。

## 前端类比

Deployment 类似一个 API service 的多个后端 origin；优先级像主站/灾备站，权重像灰度发布百分比。熔断器类似前端发现某接口持续失败后暂时不再点击重试，但它是服务端对所有新请求执行的保护。

流式首事件是 `fetch()` 已经开始消费 response body 的边界。边界后替换来源，不等于普通 HTTP 重定向，而是在同一份 UI 文本里混入另一个回答，因此禁止。

## 后果

正面结果：单个部署的短时故障可自动被备用部署吸收；429 不会被每个新请求持续撞击；客户端可审计实际部署。

代价：一次业务请求可能产生多次上游尝试；不同模型的语义与成本可能不同；内存健康状态在多实例之间不一致；随机权重不是精确的请求配额。

## 被否决的替代方案

- 所有错误都重试：会把无效请求、权限拒绝和额度拒绝放大成更多流量。
- 流式中途继续备用回答：会产生不可审计的混合答案。
- 第一版直接引入 Redis/数据库健康状态：增加跨实例协调复杂度，且尚无生产指标证明需要全局一致熔断。
- 直接让客户端选择 Provider：泄漏内部部署结构，并把灾备逻辑复制到每个应用。

## 复审触发条件

当网关多实例上线、需要跨区域路由、成本感知路由或真实健康指标时，复审健康状态存储、选择算法和探测机制。
