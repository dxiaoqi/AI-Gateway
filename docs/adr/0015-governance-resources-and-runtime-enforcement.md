# ADR 0015：统一治理资源与运行时执行

- 状态：Accepted
- 日期：2026-07-15
- 覆盖：Iteration 15–18

## 背景

模型、配额、定价、预算和安全护栏看起来是五类配置，但它们都有相同的企业要求：按租户隔离、只有授权管理员能修改、并发编辑不能互相覆盖、修改必须持久化，而且数据面必须使用最新配置。

如果每个功能独立建设 CRUD、版本和审计，代码会重复，安全边界也容易不一致。

## 决策

使用统一的 `governance_resources` 资源信封保存不同策略：

- `kind` 表示资源类型；`spec` 保存该类型的领域字段。
- `tenant_id` 是可见性和授权边界；模型部署固定使用 `*`，表示全局资源。
- `version` 配合 HTTP `If-Match` 实现乐观锁。
- `enabled` 允许停用配置而不删除历史。
- 所有创建和更新写入独立治理审计表。
- 模型配置发布后热更新进程内 Provider Registry。
- 配额、预算和护栏按请求读取当前策略，确保后台修改立即生效。
- 定价后的用量聚合写入 PostgreSQL；调用成功不因记账故障被改写成失败。

## 为什么前端可以这样理解

它类似一个通用 CMS：每篇内容都有类型、归属空间、发布状态和版本号；不同之处是这里“发布”的不是文章，而是网关真正执行的规则。

## 结果与代价

优点：权限、版本、迁移和审计行为统一，新增策略种类成本较低。

代价：`spec` 是 JSONB，数据库无法为每个领域字段提供完整列级约束，因此必须由服务层校验，并由测试保护。高频请求读取数据库也需要在规模化阶段增加带版本失效的本地缓存。

## 明确不做

- 不保存供应商 API Key 明文，只保存环境变量引用。
- 不实现物理删除，避免审计链断裂。
- 不在本轮实现多币种汇率换算、跨月结转或供应商账单对账。
- 不用正则规则代替专业 DLP/内容安全服务；当前 Guardrail 是可运行的基线。
