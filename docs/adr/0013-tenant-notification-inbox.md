# ADR 0013：事务内租户通知与独立已读回执

- 状态：Accepted
- 日期：2026-07-15
- 版本：0.13.0

## 背景

轮换申请和决策需要通知相关管理员。如果在审批请求里同步调用邮件、IM 或 Webhook，外部故障会拖慢甚至破坏核心事务；如果数据库提交后再临时发送，进程崩溃又会永久丢通知。项目也无法预设每家企业使用的通信工具。

## 决策

在与审批状态和审计相同的 PostgreSQL 事务内写入 `admin_notifications`。申请创建产生租户广播通知；终态决策产生定向原申请人的通知。已读状态单独存储在 `admin_notification_reads(notification_id, actor_id)`，不修改通知事实。

站内通知是权威事实源。外部渠道未来从该持久化事实派生投递，不进入审批请求的同步关键路径。

## 选择原因

- 状态、审计和通知要么一起成功，要么一起回滚。
- Tenant Scope 可直接进入 SQL WHERE，避免跨租户泄漏。
- 每个 actor 独立已读，符合多人审批工作台语义。
- 不绑定邮件/企业微信/Slack，保留企业部署选择。
- 通知事实不可变，read receipt 可重复 upsert，重试安全。

## 否决方案

| 方案 | 未采用原因 |
|---|---|
| 审批 API 同步发邮件/IM | 外部延迟和故障进入 Key 管理关键路径，难以保证事务一致性。 |
| 只在页面用 pending 数量推导 | 无法通知申请人批准/拒绝结果，也没有独立已读。 |
| 通知表直接加单一 `read_at` | 一个人已读会替所有人清除通知。 |
| 仅发外部消息，不存站内事实 | 无法审计是否创建过通知，也无法补偿投递。 |

## 后果

得到可靠站内通知、租户过滤和个人已读；代价是增加两张表及保留策略需求。外部投递仍需后续 outbox worker、通道签名、重试和死信设计。
