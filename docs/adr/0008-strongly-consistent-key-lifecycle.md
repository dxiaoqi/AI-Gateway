# ADR 0008：用 PostgreSQL 事务实现强一致虚拟 Key 生命周期

- 状态：Accepted
- 日期：2026-07-15
- 版本：0.8.0

## 背景

Iteration 3 从环境变量读取虚拟 Key，适合验证身份模型，但修改 Key 必须改配置并重启每个实例。企业发生人员离职、密钥泄露或权限回收时，需要确定“管理接口返回成功以后，旧 Key 是否还可能继续调用”。

## 决策

1. PostgreSQL 是虚拟 Key 的事实来源；启用数据库模式后，每次业务认证直接查询数据库，不使用进程缓存。
2. 原始 Key 只在创建或轮换响应中返回一次；数据库仅保存 `HMAC-SHA256(rawKey, pepper)`。
3. 创建、权限修改、禁用、轮换与对应审计事件写入同一数据库事务。
4. 修改与轮换必须携带 `If-Match: <version>`；版本不一致返回 409，防止后提交覆盖先提交。
5. 管理 API 使用独立 `ADMIN_BEARER_TOKEN`，业务虚拟 Key 不能调用管理 API。
6. 删除暂不开放；用 `enabled=false` 软禁用，保留身份和审计链。

## 为什么这样选

直接查库让禁用和轮换语义清楚：事务提交后的下一次认证一定读取新状态。管理变更和审计同事务避免“权限改了但没有记录”或“记录说改了但实际回滚”。版本前置条件相当于前端编辑表单时的乐观锁，能显式暴露并发冲突。

## 未选择方案

| 方案 | 未选择原因 |
|---|---|
| 继续用环境变量并滚动重启 | 吊销时间取决于部署速度，多实例容易短暂不一致 |
| 在每个实例缓存 Key | 性能更高，但需要可靠失效广播；本轮无法证明即时吊销 |
| 存储可解密原始 Key | 管理员或数据库泄露会直接暴露全部调用凭据 |
| 变更和审计分别写入 | 任一写入失败会造成事实与审计不一致 |
| 物理删除 Key | 破坏调查线索，也无法区分从未存在与曾被吊销 |

## 代价

- 每个业务请求增加一次数据库查询，数据库成为认证可用性的关键依赖。
- PostgreSQL 不可用时不降级到环境变量，否则被吊销的 Key 可能重新生效；业务请求会失败，Readiness 返回 503。
- 大规模部署需按“实例数 × 每实例连接池上限”规划连接数，必要时引入 PgBouncer/RDS Proxy。

## 后续条件

只有在引入可证明的跨实例失效机制、短 TTL 和安全故障策略后，才考虑认证缓存。未来管理员身份应从静态 Token 升级为企业 OIDC/RBAC，并把 actor 记录为真实人员/服务身份。
