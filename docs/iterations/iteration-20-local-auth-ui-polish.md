# Iteration 20：主组织注册与登录界面可用性修复

- 版本：0.20.0
- 状态：Completed
- 目标读者：领导、传统前端背景的新工程师、平台运营人员

## 30 秒领导摘要

Iteration 19 已经具备安全的账号密码能力，但新增表单没有完整继承旧登录框的 CSS，导致部分输入框显示成浏览器默认样式，密码框和“显示”按钮比例异常。本轮没有改变认证规则，只修复用户真正看到和操作的登录界面，并补齐桌面端与窄屏验收。

业务结果：第一次部署时，管理员可以清楚地完成“创建主组织 Owner”；后续访问时，同一位置会切换为普通登录表单。页面不会因为屏幕宽度或浏览器缩放而出现输入框大小不一致、按钮竖排等问题。

## 小白解释：问题为什么会发生

可以把 React 组件理解成 HTML 骨架，把 CSS 理解成装修规则。旧页面的装修规则只写了“放在 `.token-field` 房间里的输入框要长什么样”。Iteration 19 新增了组织名称、用户名和确认密码，但它们没有放进那个房间，因此浏览器只能使用默认外观。

密码输入框虽然在旧房间里，但“输入框宽度 100%”再加上右侧按钮，会超过房间总宽度。空间不足时，“显示”两个字就被挤成竖排。

## 本轮设计

1. 本地认证表单使用独立的 `.auth-field`，不再借用开发 Token 表单的结构。
2. 每个输入框都有稳定的宽度、高度、内边距、边框、聚焦状态和占位符颜色。
3. 密码框采用两列布局：左侧输入框可收缩，右侧显隐按钮保持固定宽度。
4. `label` 与 `input` 通过 `htmlFor/id` 明确关联，方便键盘操作和辅助技术识别。
5. 首次注册提示改成独立信息区，明确说明“唯一一次开放注册，完成后自动关闭”。
6. 小屏幕下密码要求换行显示，避免和字段标题互相挤压。
7. 修复本地 `127.0.0.1` 被 Next.js 规范成 `localhost` 后产生的 Origin 误判：未配置正式公开地址时，以实际 Host/代理协议判断同源；配置了 `ADMIN_CONSOLE_PUBLIC_ORIGIN` 时仍严格匹配该唯一地址。

## 验收方法

### 自动验证

```bash
npm run verify
```

### 人工页面验证

1. 打开 `http://127.0.0.1:3100/`。
2. 确认组织名称、用户名、密码、确认密码四个输入框宽度和高度一致。
3. 点击“显示/隐藏”，确认按钮不会换行，密码内容能正确切换。
4. 缩窄浏览器窗口，确认密码要求移动到下一行，表单不产生横向滚动。
5. 不提交表单，避免验收过程提前占用唯一 Owner 注册名额。

## 已知边界

- 本轮只优化未登录状态的本地账号表单，没有重新设计登录后的完整控制台。
- 浏览器密码管理器的图标由浏览器自身绘制，不属于应用 CSS 控制范围。
- 认证、密码哈希、频率限制和唯一 Owner 规则均沿用 Iteration 19，没有变更安全边界。

## 验收中发现并修复的问题

创建请求最初返回 `403 csrf_error`。浏览器的 Origin 是 `http://127.0.0.1:3100`，而 Next.js 可能把服务端 `nextUrl.origin` 表示为 `http://localhost:3100`；两者指向同一个本地服务，但字符串并不相等。修复后，本地模式使用请求实际 Host 构造期望 Origin，生产环境仍以显式配置的公开地址为准。
