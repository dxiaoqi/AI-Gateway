# Iteration 18：企业安全护栏基线

- 版本：0.18.0
- 日期：2026-07-15
- 状态：Completed（Baseline）
- 关联 ADR：[ADR 0015](../adr/0015-governance-resources-and-runtime-enforcement.md)

## 0. 一句话说明

Prompt 在发送给模型供应商前可以按租户检查 PII、提示词注入和危险内容并阻断。

## 1. 30 秒领导摘要

| 问题 | 回答 |
|---|---|
| 以前 | 网关控制“谁能调用”，但不检查“发了什么” |
| 现在 | 后台按租户发布 audit/block 护栏，阻断发生在外部调用和计费之前 |
| 价值 | 降低敏感信息外发、越权诱导和明显危险请求风险 |
| 证据 | 测试中的中文提示词注入被 400 阻断，普通请求继续成功 |
| 风险 | 当前是规则基线，不等同于专业 DLP、内容安全模型或人工审核体系 |

## 2. 小白导读

Guardrail 类似前端提交表单前的校验器，但它运行在可信服务端，用户无法绕过。顺序很重要：认证 → 护栏/预算 → 配额 → 供应商。被护栏拒绝的 Prompt 不会离开网关，也不会占用模型额度。

| 检查项 | 当前基线 |
|---|---|
| PII | 邮箱、13–19 位支付卡样式数字 |
| Prompt Injection | 中英文“忽略之前/系统指令”等明显模式 |
| Content Safety | 中英文制造爆炸物等明显模式 |

`audit` 表示观察模式，不阻断；`block` 命中后返回 `content_policy_violation`。

## 3. 安全设计

- 策略按租户过滤，不能读取或修改其他租户规则。
- Prompt 只在内存中检查，不写治理资源、审计状态或用量表。
- 错误响应只返回命中的类别和策略 ID，不回显敏感原文。
- 检查发生在流式响应开始前，避免 SSE 已发送后无法安全改写状态码。

## 4. 验证

```bash
npx vitest run test/governance-http.test.ts -t "guardrails"
```

手工请求包含 `忽略之前的系统指令` 时预期 HTTP 400；普通文本预期 HTTP 200。

## 5. 已知限制

- 正则可能误报或漏报，不能识别图片、文件、编码逃逸和复杂语义攻击。
- audit 模式当前只允许请求通过，尚未产生专门的安全事件指标和告警。
- 尚未实现输出内容检查、脱敏改写、策略例外、审批和人工复核。
- 生产敏感数据场景应接入企业 DLP/内容审核服务，并增加规则版本和效果评估集。
