# 面向领导与前端新人的记录规范

项目的主要 Review 读者有两类：

1. **领导/客户**：需要快速判断本轮解决了什么问题、有什么业务价值、风险是否可控、下一步为什么值得做。
2. **传统前端背景的新工程师**：熟悉 HTTP、JSON、Promise、React/Vue、浏览器 API，但不默认熟悉网关、模型 Provider、流控、分布式状态和后端安全。

因此每轮记录必须同时提供三个阅读深度。

## 第一层：30 秒领导摘要

必须回答：

- 以前有什么问题？
- 这一轮交付了什么可见结果？
- 对业务/团队有什么价值？
- 目前最大的剩余风险是什么？
- 如何用 2–3 条命令演示？

避免只写“新增 Adapter、Repository、HMAC”。应写成：“业务以后不需要修改代码就能切换模型；不同团队的调用可以区分和授权。”

## 第二层：前端工程师导读

优先使用已有认知建立映射：

| 网关/后端概念 | 前端类比 |
|---|---|
| Fastify Route | Next.js API Route / BFF 接口 |
| onRequest Hook | 路由守卫或 Axios interceptor |
| Provider Adapter | 对不同后端 SDK 的 service 适配层 |
| Canonical Schema | 前端内部统一 DTO/ViewModel |
| Provider Registry | 根据名称选择 service implementation |
| SSE | 浏览器 EventSource 或 fetch stream |
| Backpressure | 渲染/消费速度跟不上数据生产速度时暂停读取 |
| AbortSignal | 浏览器取消 fetch 使用的同一个 API |
| Repository | 封装 localStorage/IndexedDB/API 数据访问的 data layer |
| ACL | 前端路由权限的服务端强制版本 |
| HMAC Hash | 不保存原密码，只保存用于验证的不可逆凭据摘要 |

类比用于理解，不代替准确描述。每个关键类比后要说明差异，例如：前端隐藏菜单不是安全控制，网关 ACL 是服务端强制拒绝。

## 第三层：工程细节

必须包含：

- 一次真实请求逐步经过哪些模块。
- 每一步输入、输出和失败时的行为。
- 代码文件与职责映射。
- 成功路径和关键失败路径测试。
- 安全、并发、一致性与日志边界。
- 已知限制，不能用“后续优化”模糊带过。

## 用词规则

- 第一次出现缩写时给出中文解释，例如 ACL（访问控制列表）。
- 避免“显然”“简单”“大家都知道”。
- 不把实现动作当成业务结果。
- 每个抽象概念至少给一个具体请求例子。
- 对金钱、权限、数据泄露和可用性风险明确指出影响。
- 区分“已经自动测试”“已经真实环境验证”“仅设计未实现”。

## 汇报口径

完成一轮时，口头或聊天汇报按以下顺序：

1. “本轮已经能做到什么”。
2. “我用什么证据证明它能做到”。
3. “它现在还不能做什么”。
4. “这些限制对业务有什么影响”。
5. “下一轮优先解决什么，为什么”。
